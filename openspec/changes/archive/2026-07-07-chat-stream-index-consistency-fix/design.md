## 背景

本变更横跨会话聊天流程和知识库索引流程。聊天问题是前端生命周期问题：`SessionChat` 拥有本地 `AbortController`，但流取消仅在发送新消息前或组件卸载时触发，而未在本地聊天目标切换时触发。KB 问题跨越前端和后端边界：`KbDetail` 追踪乐观的内存中索引进度，而 `Indexer` 和 KB API 定义了文件索引状态的持久化权威来源。

当前索引实现还根据使用路径暴露了两种不同的语义。流式索引更新 `.index_status.json` 并返回分块进度，而同步索引返回集合级总数，并未完全维护相同的每文件持久化状态契约。结果是 UI 呈现的内容与后端实际保证的内容之间存在不匹配。

## 目标 / 非目标

**目标：**
- 确保过时的流式聊天响应不能在聊天切换或删除后继续改变当前活动本地聊天视图。
- 确保 KB 索引进度状态在成功和失败时都能达到终端 UI 状态。
- 对齐同步 KB 索引与流式索引，使持久化文件状态和报告的分块计数在所有入口点具有相同的含义。
- 为上述行为添加回归测试覆盖。

**非目标：**
- 不重新设计整个会话聊天架构，也不用其他传输方式替代 SSE。
- 不改变检索质量、重排序策略或 LLM 行为。
- 不将 KB 或会话持久化迁移出当前的本地文件系统模型。
- 不处理与本更改无关的仓库关注点，如包体积、CORS 策略或模型路径可移植性。

## 决策

### 决策 1：将本地聊天视图变更视为流终止点

任何使当前本地聊天目标失效的操作，必须在变更选中状态前终止正在进行的流。具体来说，切换聊天和删除当前选中的聊天将使用与开始新流之前相同的取消路径。

这保持了简单的契约：
- 一个本地页面最多有一个活跃的流
- 流只允许更新为其启动的聊天视图
- 一旦页面目标变更，旧流被视为过期且必须停止

考虑的替代方案：
- 让流保持活跃，在回调中通过检查聊天 ID 忽略延迟到达的 token。
  拒绝理由：这会将过时流的防护散布到多个回调中，并且仍然保持不必要的网络请求存活。

### 决策 2：在 SSE 事件和请求失败路径中都显式进行 KB 进度清理

KB 详情页面将把索引进度视为具有显式终端清理的状态机：
- 成功时清除每文件进度，对于批量运行还清除全局索引状态
- `index_error` 清除受影响文件的进度
- 请求级失败或抛出的 fetch 错误清除流开始前引入的任何乐观进度

这特意在页面层处理，而不仅仅在 API 辅助函数中，因为页面拥有每文件进度状态和批量运行的 `isIndexingAll` 标志。

考虑的替代方案：
- 将清理隐藏在 `kbApi.indexStream()` 中。
  拒绝理由：API 辅助函数无法安全地推理页面特定的乐观状态键或批量页面 UX 标志。

### 决策 3：将每文件索引状态定义为权威契约，在 `index_file` 层集中修复

权威索引契约将是：
- `indexed` 表示特定文件已成功完成索引
- `chunks` 表示该特定文件产生的块数
- 同步和流式索引路径必须持久化和报告相同的语义

**代码审查发现：当前两条路径都存在 `collection.count()` 问题。** 同步索引 `index_file()` 直接 `return collection.count()`（集合级），流式索引 `index_file_stream()` 虽然写入了 `set_file_status(..., chunks=collection.count())`，但同样使用了集合级计数。前者未写入状态文件，后者写入了错误的值。修复必须同时覆盖两条路径。

**修复策略：集中在 `index_file` 和 `index_file_stream` 两个底层函数修复，而非逐条路径修补。** `index_file` 是所有 4 条同步路径（单文件、文件夹、全部、上传并索引）的共同调用点。在 `index_file` 中用 `len(nodes)` 替换 `collection.count()` 并新增 `set_file_status()` 调用后，所有下游路径自动受益。同理 `index_file_stream` 是 3 条流式路径的共同点，在其中修复计数和状态持久化即可。`reindex_file` 调用 `index_file`，因此无需单独处理。

考虑的替代方案：
- 保留集合级计数，并将其记录为"当前 KB 大小"。
  拒绝理由：UI 和状态文件是以文件为导向的，而非以 KB 总量为导向，且当前命名已暗示文件本地语义。

### 决策 4：回归测试聚焦后端集成测试，前端行为通过手动验证

当前项目无前端测试基础设施（无 vitest/jest/playwright 依赖），且短时间内建立完整前端测试不划算。回归测试策略：

- **后端集成测试**（FastAPI TestClient + monkeypatch 模拟索引/检索层）：验证流取消的 API 级行为（取消的请求不产生副作用）、KB 索引进度清理的 API 可见效果（状态文件在失败后正确更新）、同步/流式索引状态一致性。
- **前端行为**：流取消和进度恢复的 UI 效果在实施后通过手动验证确认。

测试将覆盖外部可衡量的保证：
- 过时的聊天流取消后，服务器端不会产生后续副作用
- KB 索引失败时，`.index_status.json` 中的状态正确反映失败
- 同步和流式索引产生相同的持久化文件状态含义

考虑的替代方案：
- 安装 vitest + testing-library 并编写组件测试。
  拒绝理由：需额外 setup 开销（mock AbortController、fetch 等）且测试收益与成本不匹配——当前 UI 状态逻辑内嵌在 useEffect/useState 中，测试粒度粗且易因重构失效。

## 风险 / 权衡

- [风险] 添加更多显式取消行为可能暴露聊天页面中关于 `loading` 何时重置的潜在假设。
  缓解措施：保持取消处理集中化，并一起测试聊天切换、删除当前聊天和发送新消息路径。

- [风险] 重构同步索引以使用文件级计数需要修改 `index_file` 和 `index_file_stream` 两个核心函数，风险已高度集中。
  缓解措施：修改后运行全量测试套件，验证回归测试覆盖新旧行为。

- [风险] KB 进度清理代码可能在单文件索引和批量索引流程之间重复。
  缓解措施：如果需要，在页面组件内提取共享清理辅助函数，同时保持页面本地状态的拥有权在 UI 层。

## 迁移计划

1. 用回归测试锁定三个已确认问题的预期行为。
2. 更新 `SessionChat` 流生命周期处理，使聊天切换和当前聊天删除取消过时的流。
3. 修复 SSE 错误和请求失败的 KB 页面清理路径。
4. 在 `index_file` 和 `index_file_stream` 层集中修复 chunk 计数和状态持久化。
5. 运行后端回归测试，手动验证前端流取消和进度恢复行为。

## 开放问题

- 若后续在前端添加测试基础设施，可考虑将流取消和进度恢复的验证从手动提升为自动化。当前阶段不做。
