## Context

当前项目将知识库存储、会话存储、检索缓存和聊天持久化分散在多个模块中：`KnowledgeBase` 负责知识库目录与文件操作，`SessionManager` 负责会话与聊天文件，`retriever.py` 负责 BM25 与向量混合检索，前端会话页则直接消费后端返回的聊天历史与结构化来源。现状中，这几个边界缺少统一约束：路径拼接信任外部输入、BM25 缓存只按向量数量判断是否可复用、同步聊天与流式聊天持久化出的助手消息结构不一致。

这次变更是一个跨模块加固工作，涉及后端存储边界、API 契约、前端历史回放和测试策略。约束包括：尽量保持现有目录结构与公开 API 路径不变；不引入数据库迁移；在不破坏现有会话文件可读性的前提下收敛到统一的新消息格式。

## Goals / Non-Goals

**Goals:**
- 为知识库和会话相关文件操作建立统一的根目录约束，阻止路径穿越、绝对路径落盘和越界删除。
- 让 API 和内部模块对非法名称或非法目标路径返回稳定、可测试的失败行为。
- 让 BM25 检索缓存能够在知识库内容变化后正确失效，而不是继续复用陈旧文本。
- 让同步聊天与流式聊天都以相同结构持久化助手消息与来源信息。
- 让前端历史消息回放只依赖持久化数据本身恢复来源信息，而不是依赖流式期间的临时状态。

**Non-Goals:**
- 不在本次变更中引入认证、授权或多租户隔离。
- 不重构为数据库化会话存储或对象存储。
- 不改变前端整体交互布局或新增新的页面能力。
- 不修改检索排序算法本身，只解决缓存一致性与输出契约一致性。

## Decisions

### 1. 统一引入“受限路径解析”层

系统将为知识库根目录、知识库文件目录、会话根目录和聊天目录分别提供统一的受限路径解析函数。外部输入先经过语义校验，再做规范化与根目录 containment 检查；只有最终解析结果仍位于目标根目录内，才允许读写、删除或列举。

选择这个方案，是因为仅在单个调用点上做字符串过滤不可靠，容易遗漏新的入口。统一的受限路径解析函数可以同时覆盖 API、CLI 和内部调用路径，并将路径安全规则沉到同一层。备选方案是仅在 Pydantic/API 层限制输入格式，但这无法覆盖 CLI 与内部重用路径，也无法防止规范化后越界。

### 2. 标识符与文件目标分层校验

会话名、知识库名这类“标识符”采用严格规则：值在去除首尾空白后必须仍然非空；不得包含路径分隔符、`.`/`..` 段、盘符前缀、UNC 前缀和其他绝对路径语义；不得包含尾随空格或点；不得命中 Windows 保留设备名（如 `CON`、`PRN`、`AUX`、`NUL`、`COM1-9`、`LPT1-9`，大小写不敏感）；不得包含控制字符。文件目标分两类处理：聊天文件名必须是受限叶子文件名；上传文件名在 Web 上传场景下一律先裁剪为叶子文件名，再对该叶子文件名做同类合法性校验。

这样做的原因是这几类输入的业务语义不同。会话名与知识库名本身不应表达目录结构；聊天文件名由系统生成，也不需要支持嵌套路径；Web 上传如果允许客户端透传相对路径，会放大越界风险且与当前页面能力不匹配。对 Web 上传文件名采用“先裁剪叶子名，再校验叶子名”的策略，可以兼容浏览器和客户端附带路径成分的情况，同时阻止 `CON`、空白名、尾随点名这类 Windows 非法文件名进入落盘层。备选方案是对任何带路径成分的文件名一律拒绝，但这会让部分本可安全降解为叶子名的输入无谓失败。

### 3. 用“内容版本指纹”替代纯数量式 BM25 缓存失效

BM25 缓存不再只依赖 `collection.count()`。系统将根据当前可检索语料生成一个稳定的版本指纹，并以“知识库名 + 语料版本指纹”作为 BM25 缓存命中依据。语料版本可以来自 Chroma 中文档 ID/文本摘要、索引状态快照，或等价的稳定派生值，但必须在重建索引后随内容变化而变化，即使 chunk 数量不变也会失效。

选择版本指纹而不是“完全取消缓存”，是因为 BM25 重建本身有成本，项目当前仍需要保留会话级重复查询时的响应效率。选择版本指纹而不是“仅在索引入口手工清缓存”，是因为索引操作入口不止一个，且未来还可能继续扩展；以读取时的版本判断作为最终兜底更稳妥。

### 4. 统一助手消息持久化结构

同步聊天与流式聊天都将持久化为相同的消息结构：助手文本放在 `content`，结构化来源放在 `additional_kwargs.sources`。同步接口不再把来源文本拼接进持久化后的 `content` 作为唯一来源承载方式；如需返回给 CLI 或 API 调用方展示，可在响应层额外格式化，但持久化层必须统一。

选择这个方案，是因为“存储结构统一、展示格式可变”比“不同入口写不同结构”更稳。备选方案是修改前端同时兼容两种历史格式，但那只能掩盖问题，不能恢复长期一致性，也会继续增加测试复杂度。

### 5. 前端历史消息只消费结构化持久化契约

会话页在重新加载历史消息时，只从后端消息记录中的 `role`、`content` 和 `additional_kwargs.sources` 恢复界面状态。流式阶段的临时拼接仅用于当前会话中的即时渲染，刷新或切换聊天后的来源展示以持久化结果为准。

这样可以把前端状态恢复逻辑收敛到一个单一数据源。备选方案是给前端增加兼容解析逻辑，从助手文本中再反解析“来源”分段，但这会把展示层变成协议修补层，且难以稳定测试。

### 6. 对旧聊天记录采用保守兼容策略

本次变更只统一“变更后新写入”的同步聊天与流式聊天记录结构，不对历史旧记录做批量迁移，也不在前端或接口层通过正文反解析来补造结构化来源。旧同步聊天记录如果缺少 `additional_kwargs.sources`，系统只保证正文可读，来源面板按缺失处理。

选择保守兼容，是为了避免把一次边界加固工作扩展成数据迁移工程。备选方案是引入迁移脚本或惰性回填逻辑，让旧记录尽量恢复来源面板；但这会显著增加复杂度，并把来源恢复建立在脆弱的历史文本格式假设上。

## Risks / Trade-offs

- [旧会话文件中同步聊天历史没有结构化来源] → 前端需要兼容“新格式优先、旧格式仅展示正文”的降级行为，不承诺为历史旧记录自动补回来源面板。
- [路径校验收紧后，少数当前能工作但语义模糊的输入会被拒绝] → 在 API/CLI 中返回稳定、可读的错误信息，并补测试明确新边界。
- [BM25 版本指纹计算引入额外开销] → 指纹计算应复用已获取的语料元数据，避免每次查询做高成本全量重建；必要时保留进程内缓存。
- [多个模块同时修改，回归面较大] → 用单元测试覆盖路径约束、缓存失效和消息持久化一致性，用接口测试覆盖 API 契约。

## Migration Plan

1. 先引入受限路径解析与统一错误行为，在不改变 API 路径的前提下替换知识库与会话模块的直接路径拼接。
2. 再调整 BM25 缓存键与失效逻辑，确保重建索引后新查询立即命中新语料。
3. 统一同步/流式聊天持久化结构，并更新前端历史消息恢复逻辑。
4. 对旧聊天文件保持可读，不做批量迁移；新写入记录按统一结构落盘。
5. 回归验证知识库上传/删除、聊天历史加载、来源展示和重建索引后的检索结果。
