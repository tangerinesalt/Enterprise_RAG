## Context

当前项目的会话能力集中在 `SessionManager` 中，既负责会话目录和配置文件管理，也负责同步聊天与流式聊天的持久化。随着 Web UI 默认走流式接口、CLI 与部分调用方仍可使用同步接口，两个入口已经出现失败语义分叉：流式路径开始保留首问和错误消息，而同步路径仍可能留下空聊天并丢失用户问话。

与此同时，会话状态目前依赖本地 JSON 文件和 `SimpleChatStore` 落盘，写入过程普遍是“读文件 -> 改内存 -> 整体覆盖写回”。这在单用户、单请求场景下勉强可用，但一旦同一会话同时发起多次请求、切换聊天、更新配置或刷新预览，现有实现会出现配置覆盖、预览丢失、聊天内容交错等风险。

前端错误处理也已经与后端耦合在自由文本上。`SessionChat` 通过 `err.includes(...)` 区分知识库错误和模型错误，而 `/api/session/chat/stream` 的错误事件只返回 `message`。这使得文案调整、语言切换或 provider 变更都会引入静默回归。最后，当前自动化测试几乎没有覆盖这些高风险链路，现有 `tests/` 不能有效阻止上述问题反复出现。

## Goals / Non-Goals

**Goals:**
- 统一同步聊天和流式聊天的失败持久化语义，避免再出现“一个入口保留首问、另一个入口丢首问”的分叉。
- 为会话配置与聊天文件提供最小但明确的并发安全保证，优先解决同会话内的串行化写入和原子落盘。
- 将流式聊天错误事件升级为结构化协议，前端基于稳定字段分支，而不是依赖错误文案。
- 为上述行为补充自动化测试，覆盖同步、流式、错误协议和关键并发写入语义，并把这些测试作为后续实现的最低回归门槛。

**Non-Goals:**
- 不引入数据库或外部队列，仍沿用当前本地文件和 `SimpleChatStore` 方案。
- 不重新设计完整的消息状态机，也不在本次加入失败草稿、重试任务队列等扩展能力。
- 不在本次处理跨进程、多实例部署的分布式锁问题；范围限定为当前单机部署模型下的应用内串行化与原子写入。
- 不处理与本次前三点无直接关系的服务端 CORS、前端包体积或检索质量问题。

## Decisions

### Decision 1: 抽取统一的聊天执行核心，`chat()` 与 `chat_stream()` 共享持久化语义

两条聊天入口将共享同一组关键步骤：
1. 确定或创建 `chat_file`
2. 持久化用户问话与首问预览
3. 执行 KB / Chroma / 模型初始化 / 检索 / 生成
4. 成功时写入 assistant 正常回答
5. 失败时写入 assistant 错误消息

这样可以把“何时创建对话、何时写入首问、何时记录错误”从接口形态中解耦，避免同步接口继续保留旧行为。

备选方案：
- 仅修补 `chat()`：改动小，但会继续让两条路径分别演化。
- 完全删除同步接口：破坏面过大，且当前仍有现有调用方依赖。

### Decision 2: 对同一会话引入进程内锁，并将配置写入改为原子替换

本次并发安全采用“每会话一把锁”的最小方案：
- 所有会修改会话文件系统状态的路径在同一会话内串行执行，包括 `new_chat`、`select_chat`、`delete(chat_file)`、`update_config`、`chat()`、`chat_stream()` 及其预览更新逻辑。
- `config.json` 写入使用临时文件 + `os.replace()` 原子替换，避免部分写入或覆盖中断。
- 聊天文件继续通过 `SimpleChatStore` 生成完整 JSON，但外层由会话锁保护，减少并发交错。

选择它的原因是：
- 与当前文件存储架构兼容，不引入新依赖。
- 能直接降低最常见的覆盖风险。
- 复杂度明显低于数据库迁移或跨进程文件锁。

备选方案：
- 仅靠原子写入，不加锁：不能解决读改写丢更新问题。
- 全量迁移 SQLite / PostgreSQL：收益更大，但超出本次收敛范围。

### Decision 3: 流式错误事件返回结构化负载，前端只消费稳定字段

`/api/session/chat/stream` 的 `event: error` 负载将从：

```json
{"message":"..."}
```

提升为至少包含：

```json
{
  "code": "KB_INDEX_MISSING",
  "category": "kb",
  "message": "知识库 'x' 中未找到索引数据，请先在知识库中索引文件"
}
```

其中：
- `code` 用于前端分支和测试断言
- `category` 用于高层展示逻辑
- `message` 继续用于用户可见提示
- 本次最小错误码集合固定为：`KB_NOT_BOUND`、`KB_NOT_FOUND`、`KB_INDEX_MISSING`、`KB_VECTOR_EMPTY`、`MODEL_UNAVAILABLE`、`RUNTIME_ERROR`

同步接口 `/api/session/chat` 继续返回现有错误包装；本次只要求其内部失败持久化语义与流式路径对齐，不额外扩展新的结构化错误响应。

备选方案：
- 前端继续 `includes()`：实现快，但后续每次改文案都带回归风险。
- 只返回 `code` 不返回 `message`：可维护性高，但会把文案映射压力完全推给前端，不符合当前项目的简单结构。

### Decision 4: 测试按“行为回归”组织，并成为本次变更的显式能力

本次测试重点覆盖：
- 同步聊天失败时保留首问和错误消息
- 流式聊天失败时保留首问和错误消息
- SSE `error` 事件包含结构化字段
- 同一会话内并发写配置/聊天时不出现明显丢更新
- 前端错误分支改为按结构化字段工作，至少能通过类型检查与构建验证

这样能直接保护用户可见行为，同时避免把测试绑死在内部重构细节上。

## Risks / Trade-offs

- [Risk] `SessionManager` 内部继续承担较多职责，短期内不会变“优雅”  
  → Mitigation: 本次先统一语义和锁边界，后续若继续扩展再拆执行核心和存储层。

- [Risk] 进程内锁无法覆盖未来多 worker / 多进程部署  
  → Mitigation: 在 spec 和设计中明确本次保证范围是单进程应用实例；如部署模型变化，再升级为文件锁或数据库。

- [Risk] 结构化错误协议需要前后端同时升级，存在短暂兼容成本  
  → Mitigation: 前端先兼容 `message`，再切到 `code/category` 主分支；后端在过渡期保留 `message`。

- [Risk] 并发测试可能在 CI 上偶发不稳定  
  → Mitigation: 以“最终文件状态正确、不丢核心字段”为断言目标，避免依赖过于脆弱的时序假设。

## Migration Plan

1. 先补回归测试，锁定同步/流式失败语义和错误协议目标。
2. 重构 `SessionManager` 的共享聊天执行核心，使 `chat()` 与 `chat_stream()` 使用同一套持久化步骤。
3. 为会话级读改写路径引入锁，并将配置写入改为原子替换。
4. 扩展 `/api/session/chat/stream` 的错误事件负载，保留 `message` 并新增 `code/category`。
5. 更新前端 API 层和会话页错误处理逻辑，改用结构化错误字段分支。
6. 运行后端单元/集成测试与前端构建验证，并以此作为变更完成前的最小验收门槛。

回滚策略：
- 若前端未同步完成，可先保留 `message` 分支，后端结构化字段可向后兼容。
- 若会话锁实现引入死锁或明显性能问题，可临时回退到仅保留统一失败语义的变更，再单独推进并发安全。
